🎯 Motivation : Recherche Automatique

💡 Vision

Un programme CP = Modèle + Recherche. La vision (depuis ~2004) est que le solveur CP devrait être assez intelligent pour apprendre ou sélectionner automatiquement une bonne stratégie de recherche, sans que le programmeur ait à tout spécifier à la main.

C'est l'approche Black-Box : on donne le modèle au solveur, et c'est lui qui choisit comment brancher, dans quel ordre, avec quelle heuristique.

💡 Le branchement = 2 étapes : (1) sélection de variable, (2) sélection de valeur + méthode de partition. Les deux ont un impact majeur sur la taille de l'arbre de recherche !
📊

Taille de l'arbre

La variable choisie influence la profondeur et la largeur

📍

Localisation solutions

La valeur choisie détermine où les solutions se trouvent

⚡

Qualité de solution

Pour COP, le premier nœud feuille atteint affecte le B&B

🧠

Apprentissage

🔢 Heuristiques de Sélection de Valeur

🎯 Pourquoi la valeur compte ?

Pour les problèmes d'optimisation (COP), la première solution trouvée (côté gauche de l'arbre) est cruciale : elle fixe la borne pour le Branch-and-Bound. Une bonne valeur initiale = une meilleure borne = plus d'élagage = recherche plus rapide.

💡 Pour la satisfaction (CSP), la sélection de valeur influence où les solutions se trouvent dans l'arbre. Pour l'optimisation, elle influence directement la qualité de la première solution trouvée.

🏆 BIVS — Bound-Impact Value Selector

Pour chaque valeur possible de la variable sélectionnée, on calcule la borne inférieure de l'objectif après avoir fixé la variable à cette valeur. On choisit la valeur qui donne la plus petite borne inférieure (pour une minimisation).

Algorithme : Pour chaque valeur v ∈ D(x) : sauvegarder l'état → fixer x=v → propagation → mesurer min(O) → restaurer l'état. Choisir le v avec le plus petit min(O).

Complexité : O(|D(x)| × coût_propagation) — coûteux mais efficace pour trouver de bonnes premières solutions.

Conseil : utiliser BIVS uniquement pour la première solution, puis passer à une heuristique plus simple.

💾 Phase Saving

Mécanisme de mémorisation simple et efficace : si fixer x = v a réussi (n'a pas mené à un échec), on mémorise v comme "dernière valeur réussie" de x.

Au prochain branchement sur x, on essaie d'abord v. Si v n'est plus dans le domaine, on utilise l'heuristique par défaut.

// Phase Saving implémenté en MiniCP :
selectMin(q, qi -> qi.size() > 1, qi -> qi.size(), qi -> {
  int v = qi.min(); // ou la valeur sauvegardée
  return branch(() -> equal(qi, v), () -> notEqual(qi, v));
})

🔍 Heuristiques de Sélection de Variable (First-Fail)

🧩 Principe First-Fail

Le principe First-Fail : "pour réussir, essaie d'abord là où tu es le plus susceptible d'échouer". En sélectionnant la variable la plus contrainte en premier, on détecte les impasses le plus tôt possible, minimisant le travail inutile.

💡 Si toutes les variables doivent être fixées, autant détecter les impossibilités tôt ! Un échec détecté tôt dans l'arbre économise tout le sous-arbre en dessous.

📏 Min-Dom (Domaine Minimum)

Sélectionner la variable non fixée avec le plus petit domaine.

Intuition : si D(x1) = {1,2} et D(x2) = {1,…,100}, brancher sur x1 d'abord déclenche plus de propagation.

// MiniCP — N-Reines avec Min-Dom :
DFSearch search = makeDfs(cp, () -> {
  IntVar qs = selectMin(q, qi -> qi.size() > 1, qi -> qi.size());
  if (qs == null) return EMPTY;
  int v = qs.min();
  return branch(() -> equal(qs, v), () -> notEqual(qs, v));
});

🕸️ Dom+Deg et Dom/Deg

Degré d'une variable x = nombre de contraintes impliquant x. Une variable avec un degré élevé, lorsqu'elle est fixée, déclenche plus de propagations vers les autres variables.

• Dom+Deg : Min-Dom, avec ties brisés par le degré (plus grand degré = priorité).

• Dom/Deg : Sélectionner la variable avec le plus petit ratio |D(x)| / deg(x).

Heuristique combinée : variables avec petit domaine ET nombreux voisins contraints ont la priorité.

⚖️ Dom/Wdeg (Weighted Degree)

Idée : introduire de l'apprentissage. Chaque fois qu'une contrainte génère un échec (violation détectée pendant la propagation), son poids est augmenté de +1.

• Wdeg(x) = somme des poids des contraintes impliquant x ET au moins une autre variable non fixée.

• Dom/Wdeg : sélectionner la variable avec le plus petit ratio |D(x)| / Wdeg(x).

Si une contrainte C a souvent échoué par le passé, ses variables ont un Wdeg élevé → elles seront sélectionnées en priorité.

Faiblesse : la contrainte qui échoue n'est pas toujours "la seule coupable" — les cascades de propagation peuvent induire en erreur.

📉 Recherche par Impact et par Activité

🎯 Impact-Based Search (IBS)

Idée : sélectionner la décision de branchement qui réduit le moins l'espace de recherche estimé (on préfère les décisions qui "élaguent" peu, car cela signifie qu'elles sont plus susceptibles de mener à une solution).

Définitions :

• S(P) = ∏ₓ |D(x)| = estimation de la taille de l'arbre de recherche

• I(x=a) = 1 − S(P|ₓ₌ₐ) / S(P) = impact de fixer x=a (∈ [0,1], 1 = échec)

• Ī(x=a) = moyenne de I(x=a) observée sur les nœuds passés (avec facteur d'oubli possible)

Impact d'une variable : I(x) = Σ_{a ∈ D(x)} (1 − Ī(x=a))
Heuristique variable : sélectionner x avec minimum I(x)
Heuristique valeur : sélectionner a avec minimum Ī(x=a)

Mise à jour avec oubli exponentiel (α ∈ [0,1]) :
Ī_k(x=a) = (1−α)·Ī_{k-1}(x=a) + α·I^k(x=a)

Initialisation : quelques "random dives" avant la recherche complète pour estimer les impacts initiaux.

📈 Activity-Based Search (ABS)

Idée : tracer à quelle fréquence le domaine d'une variable est contracté lors de la recherche. Une variable dont le domaine est souvent réduit est un "nœud chaud" → la brancher tôt réduit la profondeur de l'arbre.

• A(x) = compteur d'activité de x, incrémenté à chaque contraction de D(x) pendant la propagation.

• Heuristique : sélectionner x avec le rapport |D(x)| / A(x) le plus petit (ou A(x) le plus grand).

Comme IBS, ABS est initialisé avec des dives aléatoires pour estimer les activités de base.

✅ Avantages IBS/ABS

• Apprentissage pendant la recherche
• Capture les corrélations entre variables
• Robuste sur des problèmes variés
• Indépendant du modèle

⚠️ Inconvénients IBS/ABS

• Coût d'initialisation élevé
• Impact peut être mal estimé
• Mémoire supplémentaire nécessaire
• Plus complexe à implémenter

⚔️ Heuristiques Basées sur les Conflits

🔴 Last-Conflict Search

Observation : si brancher sur la variable x a causé un échec récemment, elle est susceptible de causer un échec à nouveau (peu de choses ont changé). Donc, on lui donne la priorité.

nextVarToBranchOn(X) {
  if (lastConflictVariable == null)
    return fallBackHeuristic(X)
  else
    return lastConflictVariable // la dernière variable à avoir échoué
}

Quand brancher sur lastConflictVariable mène à succès (fixation réussie), on revient à l'heuristique de base.

⏱️ Conflict-Ordering Search (COS)

Généralisation de Last-Conflict : chaque variable reçoit un timestamp correspondant au nœud d'échec le plus récent où elle a causé une défaillance.

• Un compteur global nConflicts est incrémenté à chaque nœud d'échec.

• stamp[x] = nConflicts au dernier nœud où brancher sur x a échoué.

• COS : sélectionner la variable non fixée avec le plus grand timestamp. Si aucune variable n'a encore causé d'échec, utiliser un heuristique de repli.

// COS en pseudo-code :
failed = { x ∈ X : stamp[x] > 0 && |D(x)| > 1 }
if (failed est vide)
  lastVar = varHeuristic.select() // heuristique de repli
else
  lastVar = argmax_{x ∈ failed} { stamp[x] } // plus grand timestamp

COS est particulièrement efficace quand un petit sous-ensemble de variables cause la plupart des conflits (ex. AllDifferent sur un sous-groupe).

Exemple de l'utilité de COS :

Pour un CSP avec variables x1,…,x8 où le sous-ensemble {x5,x6,x7,x8} cause un conflit AllDifferent : avec l'ordre statique [x1,…,x8], on explore un grand arbre avant de détecter le conflit. COS apprend rapidement à brancher d'abord sur x5,x6,x7,x8, réduisant dramatiquement l'arbre de recherche.

🔀 Discrepancy Search (Recherche par Discordance)

🧭 Idée

La discordance = nombre de fois où on choisit la mauvaise branche (celle que l'heuristique ne recommande pas). Si on fait confiance à notre heuristique de valeur, les solutions de bonne qualité devraient être à faible discordance.

Stratégie : explorer d'abord les nœuds avec discordance 0 (suivre toujours l'heuristique), puis discordance 1 (une "mauvaise" décision), puis 2, etc.

💡 Ceci est un search combinator : il enveloppe n'importe quel branchement existant et réorganise l'ordre d'exploration sans changer le modèle.
📐 Implémentation avec LimitedDiscrepancy

On utilise une boucle sur le niveau de discordance maximal autorisé :

Objective obj = cp.minimize(totCost);

for (int dL = 0; dL < x.length; dL++) {
    DFSearch dfs = makeDfs(cp, limitedDiscrepancy(firstFail(x), dL));
    dfs.optimize(obj);
}
// dL=0 : suivre toujours l'heuristique (0 erreur)
// dL=1 : au plus 1 "mauvaise" décision autorisée
// dL=2 : au plus 2, etc.

LimitedDiscrepancyBranching est un wrapper : il filtre les alternatives qui dépasseraient maxD discordances.

Quand utiliser Discrepancy Search ?

• Quand on a une bonne heuristique de valeur (proche de la solution optimale).

• Quand les erreurs se produisent surtout en début d'arbre (nœuds peu profonds).

• En combinaison avec First-Fail pour la variable : on fait confiance à l'heuristique de valeur pour guider, FF pour éviter les échecs.

🔄 Restarts et No-Goods

🔄 Stratégie de redémarrage

Deux observations justifient les restarts :

1. Les erreurs précoces (en début d'arbre) sont très coûteuses : il faut remonter tout en haut pour les corriger.

2. Les heuristiques avec apprentissage (IBS, ABS, Wdeg) se comporteraient différemment depuis le début si elles avaient déjà appris quelque chose.

💡 Corollaire : recommencer la recherche depuis zéro, mais avec l'heuristique "plus sage" qui a appris de ses erreurs passées + une limite sur le nombre d'échecs (séquence de Luby).
📐 Séquence de Luby

La séquence de Luby définit les limites d'échecs avant chaque restart :

S = (1, 1, 2, 1, 1, 2, 4, 1, 1, 2, 1, 1, 2, 4, 8, 1, …)

Cette séquence est optimale pour les algorithmes Las Vegas. Elle garantit un speedup polynomial par rapport à la meilleure exécution unique.

🚫 No-Goods (Pour ne pas re-explorer)

Quand on redémarre, on risque de réexplorer les mêmes parties de l'arbre. Les no-goods permettent d'éviter ça :

Si la recherche a échoué avec les assignations x₂=2 ∧ x₁=6 ∧ x₅=2, on ajoute la contrainte :

x₂ ≠ 2 ∨ x₁ ≠ 6 ∨ x₅ ≠ 2 (no-good)

Ces no-goods s'accumulent au fil des restarts, empêchant de revisiter les mêmes sous-espaces. C'est une forme d'apprentissage entre redémarrages.

Distributions Heavy-Tailed

Les temps de résolution CP suivent souvent des distributions "à queue lourde" : certaines exécutions sont très courtes, d'autres très longues (les moments sont infinis !). En ajoutant de la randomisation dans les heuristiques et en faisant des restarts, on tire parti des courtes exécutions et on évite les longues — même sans apprentissage explicite.
