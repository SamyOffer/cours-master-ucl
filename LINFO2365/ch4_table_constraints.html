📋 Qu'est-ce qu'une Contrainte de Table ?

🧒 C'est quoi une Table ?

Imagine un menu de restaurant avec des règles de combinaisons : certaines entrées vont avec certains plats et certains desserts, d'autres non. La contrainte de Table, c'est exactement ça : une liste de combinaisons autorisées (les tuples).

Par exemple, pour 3 variables (x, y, z) qui prennent des valeurs dans {0,1,2}, une table peut dire : les seules combinaisons valides sont (0,1,2), (1,0,2), (2,1,0). Toutes les autres sont interdites.

Avantages : très général — on peut exprimer n'importe quelle relation en énumérant ses solutions. Parfait pour des règles complexes difficiles à formuler avec des inégalités.

💡 Table = définition en extension de la contrainte. On liste TOUTES les combinaisons valides.

🔍 Filtrage : Retirer les Valeurs Sans Support

🧒 C'est quoi "avoir un support" ?

Une valeur v pour la variable xᵢ a un support s'il existe au moins un tuple valide dans la table qui assigne la valeur v à xᵢ ET qui est cohérent avec les domaines actuels des autres variables.

Analogie : tu cherches un appartement. Une contrainte de table dit : "les seules combinaisons (quartier, taille, prix) que tu acceptes sont celles de la liste". Si tous les appartements de ton quartier préféré dans la liste sont maintenant hors budget (domaine de prix réduit), ce quartier n'a plus de support → il est retiré du domaine.

💡 Filtrage table = supprimer toute valeur qui n'est soutenue par aucun tuple encore compatible.

  • Pour chaque variable xᵢ et chaque valeur v ∈ D(xᵢ) :

  • Chercher un tuple t dans la table tel que t[i] = v ET pour tout j ≠ i, t[j] ∈ D(xⱼ)

  • Si aucun tel tuple n'existe → retirer v de D(xᵢ)

  • Répéter jusqu'au fixpoint (aucune valeur retirée)

⚡ Compact Table (CT) — Filtrage Efficace avec BitSets

🧒 C'est quoi le Compact Table ?

L'approche naïve vérifie chaque valeur de chaque variable contre tous les tuples → très lent pour de grandes tables. Compact Table (CT) utilise des masques de bits pour accélérer énormément.

Idée : Représente l'ensemble des tuples valides comme un bitset — un nombre binaire où le bit i vaut 1 si le tuple i est encore possible. Pour filtrer, on fait des opérations AND sur des bitsets pré-calculés.

Analogie : Imagine que chaque tuple est une lumière (allumée = valide, éteinte = invalide). Pour savoir quels tuples restent valides si x=0, tu allumes seulement les lumières des tuples où x=0. Tu fais ça pour toutes les variables, et tu fais un ET logique → seules les lumières allumées partout restent.

💡 CT réduit le filtrage à des opérations bitwise (AND, OR) qui sont ultra-rapides sur les processeurs modernes.
📐 Algorithme CT

📌 "Le masque de tuples supportés = intersection des masques de chaque valeur dans son domaine"

TermeSignification
mask[i][v]BitSet : bit j = 1 si le tuple j a la valeur v pour la variable xᵢ
∪ sur D(xᵢ)OR de tous les masques des valeurs encore dans le domaine de xᵢ
∩ sur iAND de tous les masques pour toutes les variables
supTuplesBitSet final : bit j=1 ssi le tuple j est encore compatible

Filtrage : La valeur v de la variable xᵢ peut être retirée si (supTuples AND mask[i][v]) = 0 — c'est-à-dire qu'aucun tuple compatible n'utilise la valeur v pour xᵢ.

📊 Comparaison des Performances

🧒 Pourquoi CT est-il meilleur ?

L'algorithme naïf a une complexité de O(t × n × d) où t = nombre de tuples, n = nombre de variables, d = taille max des domaines. CT réduit ça à O(t/w × n × d) où w est la taille d'un mot machine (ex: 64 bits) — un facteur 64 de gain !

💡 Le gain en pratique peut être de 10x à 100x selon la taille de la table.
