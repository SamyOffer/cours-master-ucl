🔧 Le Problème de Job Shop

🧒 C'est quoi le Job Shop ?

Le Job Shop est un problème d'ordonnancement classique : plusieurs jobs doivent être traités par plusieurs machines, dans un ordre précis. Chaque machine ne peut traiter qu'un seul job à la fois (ressource disjonctive, capacité 1).

Exemple : Une usine avec 3 machines M1, M2, M3, et 3 jobs. Job 1 nécessite M1 puis M2 puis M3 dans cet ordre, avec des contraintes de précédence entre tâches d'un même job.

💡 La contrainte Disjunctive est un cas particulier de Cumulative avec toutes les demandes rᵢ = 1 et une capacité C = 1.
🏭

Activités

Chaque tâche a un début s[i], durée d[i], fin e[i] = s[i]+d[i]

🔴

Ressource unaire

Une machine = une ressource de capacité 1 (une tâche à la fois)

➡️

Précédences

Les tâches d'un même job doivent s'exécuter dans l'ordre imposé

🎯

Makespan

Objectif : minimiser la durée totale = maximum des dates de fin

📐 Modèle MiniCP du Job Shop

Chaque machine reçoit une contrainte Disjunctive sur ses activités :

// Variables de début et de fin pour chaque tâche
IntVar[][] start = new IntVar[nJobs][nMachines];
IntVar[][] end   = new IntVar[nJobs][nMachines];

// Contraintes de précédence entre tâches d'un même job
for (int i = 0; i < nJobs; i++)
  for (int j = 1; j < nMachines; j++)
    cp.post(lessOrEqual(end[i][j-1], start[i][j]));

// Contrainte disjonctive sur chaque machine
for (int m = 0; m < nMachines; m++) {
  IntVar[] start_m = collect(start, m);
  int[]    dur_m   = collect(duration, m);
  cp.post(new Disjunctive(start_m, dur_m));
}

// Objectif = minimiser le makespan
IntVar makespan = maximum(endLast);
cp.minimize(makespan);

🔀 Décomposition Binaire

🧩 Idée de base

Pour un ensemble T de n activités non-chevauchantes, on peut imposer que pour toute paire (i,j) avec i < j, soit i se termine avant j, soit j se termine avant i :

∀ i, j ∈ T avec i < j : bᵢⱼ ≠ bⱼᵢ

où bᵢⱼ est une variable booléenne réifiée : bᵢⱼ = 1 signifie "i finit avant que j commence".

⚠️ La décomposition binaire avec contraintes réifiées est au moins aussi forte que le filtrage TimeTable du Cumulatif, mais elle génère O(n²) contraintes.

Pourquoi la décomposition binaire peut être plus forte ?

Exemple : si l'activité A n'a pas de partie obligatoire, le filtrage TimeTable ne peut pas pousser le début de B. Mais la décomposition binaire peut détecter que A ≪ B est forcé.

Recherche sur les variables de booléens : On peut brancher sur les variables bᵢⱼ (l'ordre entre chaque paire) plutôt que sur les dates de début. Cela fixe d'abord l'ordonnancement sur chaque machine, puis les dates de début.

🔍 Deux stratégies de recherche pour le Job Shop

Stratégie 1 : Fixer les dates de début

On choisit une activité non fixée (ex. B), on branche sur une valeur de son début. Plus direct, mais peut conduire à beaucoup de retours arrière.

Stratégie 2 : Fixer l'ordre (recommandée)

On branche sur les variables bᵢⱼ. Une fois l'ordre total fixé sur chaque machine, le makespan minimum est calculable directement (toujours faisable).

⏱️ Earliest Completion Time (ECT)

🎯 Notion clé

Pour un ensemble Ω d'activités non-chevauchantes, on définit :

• estΩ = min{estⱼ | j ∈ Ω} = plus tôt début possible de l'ensemble

• lctΩ = max{lctⱼ | j ∈ Ω} = plus tard fin possible de l'ensemble

• dΩ = Σⱼ dⱼ = durée totale de l'ensemble

• ectΩ = borne inférieure sur la date de fin au plus tôt de l'ensemble

💡 Si A, B, C, D doivent tous précéder E, alors E ne peut pas commencer avant ect({A,B,C,D}). Cette propriété permet de filtrer le début de E !
📐 Calcul de ECT en O(n log n)

Borne inférieure sur l'ECT d'un ensemble Ω :

ectΩLB = max{ estΩ' + dΩ' | Ω' ⊆ Ω }

Algorithme glouton (tri par est, puis greedy) :

ComputeECTLowerBound(T) {
  T_sorted ← sortAZ(T, sortKey = est) // O(n log n)
  ect = -∞
  for i ← T_sorted {
    ect ← max(estᵢ + dᵢ, ect + dᵢ)
  }
  return ect
}

📌 Calculer ECT pour tous les sous-ensembles en O(n²) serait trop coûteux → on utilise la structure Θ-Tree pour le faire en O(n log n).

Complexité NP-difficile !

Le calcul exact de l'ECT pour des activités avec fenêtres de temps est NP-difficile (réduction depuis le problème 3-Partition). La borne inférieure vue ci-dessus est donc cruciale : elle est calculable en O(n log n) tout en restant utile pour le filtrage.

🌲 Structure Θ-Tree

🔢 Structure de données avancée

Le Θ-Tree est un arbre binaire équilibré permettant de calculer efficacement l'ECT d'un ensemble d'activités et ses sous-ensembles imbriqués.

C'est un arbre binaire équilibré dont les feuilles correspondent aux activités (triées par est), et dont les nœuds internes maintiennent des valeurs Δ et ect intermédiaires.

💡 La racine du Θ-Tree contient directement l'ECT de l'ensemble complet des activités insérées !
⚡ Complexités du Θ-Tree
OpérationComplexitéDescription
init({1..n})O(n log n)Initialiser un Θ-Tree vide pour n activités
insert(i)O(log n)Insérer l'activité i dans le Θ-Tree
remove(i)O(log n)Retirer l'activité i (set Δᵢ=0, ectᵢ=−∞)
ectO(1)Lire l'ECT de l'ensemble courant (à la racine)

Pourquoi a-t-on besoin du Θ-Tree ?

Pour les algorithmes de filtrage (Overload Checking, Detectable Precedences, Not-Last), on doit calculer l'ECT de nombreux sous-ensembles imbriqués d'activités. Sans Θ-Tree, chaque calcul prend O(n log n), donc l'algorithme total prendrait O(n² log n). Avec le Θ-Tree et son insertion/suppression en O(log n), l'algorithme complet s'exécute en O(n log n).

Chaque nœud interne du Θ-Tree stocke :

• Δ = durée totale des activités dans le sous-arbre

• ect = ECT de l'ensemble des activités dans le sous-arbre

Ces valeurs se mettent à jour en remontant vers la racine après chaque insertion/suppression.

🚨 Overload Checking (Vérification de surcharge)

✅ Règle de faisabilité

L'Overload Checking est une règle de faisabilité : si un sous-ensemble d'activités ne peut pas toutes s'exécuter dans leurs fenêtres de temps, on détecte un échec.

Règle : ∀ Ω ⊆ T : (estΩ + dΩ > lctΩ) → ÉCHEC

Si les activités d'un sous-ensemble Ω ont une durée totale qui dépasse leur fenêtre de temps commune, c'est infaisable !

⚠️ Cette règle n'est pas capturée par la décomposition binaire seule — elle nécessite une vue globale de l'ensemble des activités.
⚡ Algorithme efficace en O(n log n)

Reformulation : au lieu de tester tous les 2ⁿ sous-ensembles, on utilise les "Left Cuts" :

LCut(T,j) = { i | i ∈ T & lctᵢ ≤ lctⱼ }

La condition se reformule en : ∀ j ∈ T : ectLCut(T,j) > lctⱼ → ÉCHEC

OverloadCheckEfficient(T={1..n}) {
  T ← sortAZ([1..n], sortKey = lct) // O(n log n)
  Θ ← Θ-Tree.init({1..n}) // O(n log n)
  for (j ← T) {
    Θ.insert(j) // O(log n) — invariant: Θ contient LCut(T,j)
    if (Θ.ect > lctⱼ) { // O(1)
      throw InconsistencyException // ÉCHEC !
    }
  }
}

Complexité totale : O(n log n) — on trie une fois, puis on fait n insertions et n vérifications.

Intuition : On trie les activités par lct croissant. En insérant les activités une par une, on construit progressivement LCut(T,j) de façon incrémentale. Le Θ-Tree nous donne ect de l'ensemble courant en O(1). Si ect > lctⱼ, c'est un surcharge : fail !

🔗 Detectable Precedences (Précédences Détectables)

🧩 Règle de filtrage

Les Detectable Precedences sont une règle de filtrage : si on peut déduire que certaines activités doivent précéder l'activité i (parce qu'elles ne peuvent pas se placer après), on peut pousser le début de i.

Une précédence j ≪ i est détectable si :

estᵢ + dᵢ > lctⱼ − dⱼ  (c.-à-d. ectᵢ > lstⱼ)

Autrement dit : j ne peut pas commencer après que i se termine, donc j doit précéder i.

💡 Filtrage : pour chaque i ∈ T, on pousse estᵢ ← max(estᵢ, ectDPrec(T,i)) où DPrec(T,i) est l'ensemble des activités détectées comme devant précéder i.
⚡ Algorithme en O(n log n)

On trie les activités par ect croissant pour avoir des ensembles DPrec imbriqués :

DetectablePrecedence(T={1..n}) {
  Tlst ← sortAZ([1..n], sortKey = lct−d) // O(n log n)
  Tect ← sortAZ([1..n], sortKey = est+d) // O(n log n)
  ite ← iterator(Tlst)
  j ← ite.next() // candidat précédant i
  Θ ← Θ-Tree.init({1..n}) // O(n log n)
  for (i ← Tect) {
    while (estᵢ + dᵢ > lctⱼ − dⱼ) {
      Θ.insert(j) // O(log n)
      if (ite.hasNext()) j ← ite.next()
      else break
    }
    // Θ contient DPrec'(T,i) incluant i
    Θ.remove(i) // O(log n)
    est'ᵢ ← max(estᵢ, Θ.ect) // O(1)
    Θ.insert(i) // O(log n)
  }
  estᵢ ← est'ᵢ, ∀i ∈ T // mise à jour simultanée
}

Pourquoi les ensembles DPrec sont-ils imbriqués ?

En triant les activités i par ordre croissant de estᵢ + dᵢ (= ectᵢ), les ensembles DPrec'(T,i) forment une séquence croissante : DPrec'(T,1) ⊆ DPrec'(T,2) ⊆ … ⊆ DPrec'(T,n). Cela permet d'insérer les activités dans le Θ-Tree de façon incrémentale sans jamais avoir à tout reconstruire.

🚫 Not-Last (Règle "Pas en Dernier")

🔄 Règle symétrique

La règle Not-Last est le symétrique des Detectable Precedences : si un ensemble Ω d'activités ne peut pas toutes se placer après i (car elles rempliraient le temps disponible), alors i ne peut pas être la dernière activité.

Règle (NL) : ∀ Ω ⊂ T non-vide, ∀ i ∈ T\Ω :

estΩ + dΩ > lctᵢ − dᵢ → lctᵢ ← min(lctᵢ, max{lctⱼ − dⱼ | j ∈ Ω})

💡 On filtre la date de fin au plus tard de i (on la réduit). La règle relaxée NL' est équivalente au fixpoint : ectNLSet(T,i) > lctᵢ − dᵢ → lctᵢ ← max{lctⱼ − dⱼ | j ∈ NLSet(T,i)}
📐 NLSet et imbrication

NLSet(T,i) = { j | j ∈ T\{i} & lctⱼ − dⱼ < lctᵢ }

En triant les activités i par lct croissant, les ensembles NLSet'(T,i) sont imbriqués → on peut utiliser un Θ-Tree de façon incrémentale.

Complexité totale de l'algorithme Not-Last : O(n log n)

📋 Récapitulatif — Filtrage Disjonctif

Les 3 règles de filtrage pour Disjunctive, toutes en O(n log n) via Θ-Tree :

1. Overload Checking : détecte l'infaisabilité si ectLCut(T,j) > lctⱼ pour un j quelconque.

2. Detectable Precedences : filtre estᵢ (borne inférieure) en utilisant DPrec(T,i).

3. Not-Last : filtre lctᵢ (borne supérieure) en utilisant NLSet(T,i).
