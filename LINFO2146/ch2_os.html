💾 Systèmes d'exploitation pour WSN & IoT

TinyOS, Contiki, RIOT, FreeRTOS — les OS pour appareils contraints

🤔 1. Pourquoi un OS pour les systèmes embarqués ?

🧒 C'est quoi un OS embarqué ?

Sur ton PC, Windows ou Linux gère la mémoire, les programmes, les fichiers... C'est un OS. Sur un capteur IoT de 8KB de RAM, on ne peut pas mettre Windows ! Il faut un OS ultra-léger et spécialisé.

En théorie, on pourrait même programmer directement sans OS — c'est ce que les ingénieurs faisaient pendant des années. Mais c'est fastidieux : il faut tout gérer soi-même (interruptions, mémoire, scheduler...).

Un OS embarqué = couche logicielle minimale qui gère le matériel et permet à plusieurs tâches de coexister, le tout dans quelques kilobytes.

Pourquoi pas Linux ?

📌 Problème avec Linux sur WSN

Linux est possible sur des appareils avec assez de mémoire (Raspberry Pi), mais :

  • Linux nécessite une MMU (Memory Management Unit) pour la protection mémoire
  • Le noyau Linux ne peut pas être interrompu par des ISR utilisateur → latence > 100ms !
  • Pour l'embarqué temps-réel, ça ne suffit pas. Un airbag doit déclencher en <1ms !

Solution : un RTOS (Real-Time OS) ou un OS spécialisé IoT.

🔬 2. TinyOS — Le pionnier

🧒 C'est quoi TinyOS ?

TinyOS est le premier grand OS pour capteurs. Né à Berkeley en 1999, il est tellement léger que son noyau tient dans 400 octets — moins qu'un email !

Son idée clé : le programme ne tourne pas en boucle. Il attend des événements (un bouton pressé, un paquet reçu, un timer qui sonne) et réagit. Quand il n'y a rien à faire, le CPU dort.

TinyOS = modèle événement/tâche. Haute priorité pour les événements (interruptions), basse priorité pour les tâches. Pas de préemption entre tâches → file FIFO.

Modèle d'exécution TinyOS

📌 Événements vs Tâches
  • Événements (haute priorité) : déclenchés par les interruptions matérielles (bouton, radio, timer). Peuvent interrompre les tâches.
  • Tâches (basse priorité) : traitement long en arrière-plan. Attendent dans une file FIFO. Ne s'interrompent pas entre elles.
// Exemple TinyOS : clignoter une LED toutes les 500ms implementation { // Appelé au démarrage event void Boot.booted() { call MyTimer.startPeriodic(500); // 500ms } // Appelé à chaque fois que le timer sonne event void MyTimer.fired() { call Leds.led0Toggle(); // bascule la LED post myLongTask(); // poste une tâche longue } task void myLongTask() { // traitement long (calcul, écriture SD...) } }

Synchrone vs Asynchrone (NesC)

⚠️ Problème de concurrence

Si un événement peut interrompre une tâche, il peut modifier une variable partagée en même temps → race condition !

NesC force l'utilisation de sections atomiques :

atomic { // désactive les interruptions sharedCounter++; // accès sécurisé } // réactive les interruptions

Le compilateur NesC peut détecter certaines races conditions par analyse statique.

🔄 3. Contiki — L'OS coopératif

🧒 C'est quoi Contiki ?

Contiki (2004, SICS Suède) prend une approche différente : au lieu d'événements, il utilise des processus coopératifs. Pense aux coroutines : un processus s'exécute jusqu'à ce qu'il décide de faire une pause et de laisser la main aux autres.

Contrairement à TinyOS, Contiki utilise du C standard (pas un langage spécial). Les applications IoT modernes utilisent souvent Contiki-NG (la version actuelle).

Contiki = processus coopératifs (non-préemptifs) implémentés via des Proto-Threads. Un processus ne peut être interrompu que s'il attend un événement.
// Exemple Contiki : Hello World PROCESS(hello_world_process, "Hello world"); AUTOSTART_PROCESSES(&hello_world_process); PROCESS_THREAD(hello_world_process, ev, data) { PROCESS_BEGIN(); printf("Hello, world\n"); // Attendre un événement (ex: bouton pressé) PROCESS_WAIT_EVENT_UNTIL( (ev == sensors_event) && (data == &button_sensor) ); printf("Bouton pressé !\n"); PROCESS_END(); }

Scheduling coopératif

📌 Coopératif vs Préemptif
  • Coopératif (Contiki) : un processus s'exécute jusqu'à ce qu'il appelle PROCESS_WAIT_EVENT. Pas de préemption forcée. Simple, mais un processus "égoïste" peut bloquer les autres.
  • Préemptif (RTOS classique) : l'OS peut interrompre un processus à tout moment pour en lancer un autre de plus haute priorité. Plus complexe mais plus réactif.

🌐 4. Autres OS IoT

🧒 Lequel choisir ?

Il n'y a pas d'OS universel pour l'IoT — le choix dépend du matériel, des ressources disponibles, et des besoins temps-réel. Voici les principaux :

⚡ FreeRTOS

Le plus utilisé dans l'industrie. Supporte des centaines de microcontrôleurs. Préemptif, avec priorités. Utilisé dans AWS IoT Greengrass.

🚀 RIOT

OS moderne pour IoT, conçu autour des standards Internet (IPv6, 6LoWPAN). Micro-kernel modulaire. Supporte ARM, MSP430, x86.

⚙️ Zephyr

Projet Linux Foundation. Très complet (Bluetooth, WiFi, Thread). Soutenu par Intel, Nordic Semiconductor. Croissance rapide.

🦅 VxWorks

RTOS commercial. Utilisé dans les systèmes critiques (avions, Mars Rovers NASA). Context switch prévisible, POSIX threads. >100KB.

🧵 5. Proto-Threads — L'astuce de Contiki

🧒 C'est quoi un Proto-Thread ?

Un thread normal (Linux/Windows) utilise sa propre pile mémoire (stack). Sur un capteur avec 8KB de RAM, on ne peut pas se permettre d'avoir plusieurs stacks !

Un Proto-Thread est un thread léger qui partage la même pile. L'astuce : il utilise des macros C spéciales (PROCESS_WAIT_EVENT) qui mémorisent la position dans le code pour y revenir plus tard — sans avoir besoin d'une pile dédiée.

💾
Thread normal
~1-8 KB de stack par thread
🪶
Proto-Thread
Quelques octets par processus !
Proto-Thread = coroutine en C. Partage la pile principale. Idéal pour les appareils avec très peu de RAM.

⚡ 6. Interruptions : ISR, IRQ et IVT

🧒 C'est quoi une interruption ?

Imagine que tu lis un livre et que ton téléphone sonne. Tu as deux options : vérifier ton téléphone toutes les 5 secondes (polling) ou attendre que ça sonne et réagir (interruption). Les microcontrôleurs IoT préfèrent les interruptions pour économiser l'énergie !

Une interruption = un signal matériel qui interrompt le CPU pour qu'il exécute immédiatement une routine spéciale, puis reprend ce qu'il faisait.

Les trois acronymes clés

AcronymeSignificationRôle
ISR Interrupt Service Routine La fonction appelée lorsqu'une interruption se déclenche. Code court, exécuté en priorité maximale.
IRQ Interrupt ReQuest Le signal électrique envoyé par le matériel (bouton, timer, radio…) pour demander l'attention du CPU.
IVT Interrupt Vector Table Table en mémoire qui mappe chaque IRQ à l'adresse de son ISR correspondant. Le CPU s'y réfère dès qu'une interruption arrive.
📌 Comment ça marche ensemble ?
  1. Un bouton est appuyé → génère un signal IRQ vers le CPU
  2. Le CPU consulte l'IVT pour trouver l'adresse de l'ISR associée à ce type d'interruption
  3. Le CPU sauvegarde son contexte (registres, PC) et saute à l'ISR
  4. L'ISR s'exécute (lire le bouton, allumer une LED…)
  5. L'ISR termine → CPU restaure son contexte et reprend le programme principal

Polling vs Interruptions — Comparaison

🧒 Pourquoi les interruptions sont meilleures pour l'IoT ?

En polling, le CPU vérifie en boucle si quelque chose s'est passé — il ne peut jamais dormir. Avec les interruptions, le CPU dort jusqu'à ce qu'un événement arrive. Sur batterie, c'est crucial !

Caractéristique🔄 Polling⚡ Interruptions
Activité CPUToujours actif (busy-waiting)Peut dormir entre les événements
Consommation énergie❌ Élevée — gaspille les cycles CPU✅ Faible — CPU se réveille à la demande
Complexité code✅ Simple à programmer⚠ Nécessite ISR + IVT configurés
Timing✅ Prévisible (vérification régulière)⚠ Asynchrone (déclenché par événement)
Hardware spécial❌ Non (juste une boucle)✅ Oui (contrôleur d'interruptions)
Cas d'usage IoTRare (capteurs très rapides)✅ Très courant (boutons, radio, timers)
🔋 Impact énergétique

Un microcontrôleur MSP430 en polling consomme ~1 mA. En mode sleep avec interruptions, il consomme ~1 µA — soit 1000× moins ! Sur batterie AA (2000 mAh) :

  • Polling continu → batterie dure ~83 jours
  • Sleep + interruptions → batterie dure ~228 ans !

🔄 7. Synchrone vs Asynchrone — Événements TinyOS

🧒 Appels synchrones vs asynchrones

Quand tu appelles une fonction en C normal, tu attends qu'elle finisse : c'est synchrone. En IoT, attendre est un luxe (énergie, latence). TinyOS utilise asynchrone : tu demandes quelque chose et tu repasses en attente, puis tu es notifié quand c'est fini.

Synchrone (bloquant)

📌 Synchrone = appel bloquant

result = send(msg); // Attend la transmission, puis revient

  • Sender et receiver sont tous deux actifs au même moment
  • La fonction ne revient que quand l'action est terminée
  • Plus facile à comprendre (comme les appels de fonction "normaux")
  • ❌ Problème en IoT : Si le radio est lent, le CPU attend et consomme (mauvais pour batterie)

Asynchrone (split-phase, non-bloquant)

📌 Asynchrone = appel immédiat + notification ultérieure

call Radio.send(msg); // Revient IMMÉDIATEMENT
...
event Radio.sendDone(msg, error) { ... } // Appelé plus tard

  • L'appel revient immédiatement, sans attendre
  • Quand l'action est finie, une callback (événement) est appelée
  • ✅ Le CPU peut faire d'autres choses (ou dormir) pendant ce temps
  • ✅ Idéal pour IoT : économise l'énergie, permet la concurrence
  • ❌ Plus complexe à programmer (états et callbacks partout)

TinyOS : split-phase operations

🧒 Pourquoi TinyOS force l'asynchrone

TinyOS a choisi d'interdire les appels synchrones bloquants. Chaque opération longue (I/O, radio, timer) est split-phase : une fonction qui revient vite, et un événement qui te dit quand c'est fini.

Exemple : Envoi par radio en TinyOS

// Interface (ce que tu peux faire)
command error_t Send.send(message_t* msg, uint8_t len);
event void Send.sendDone(message_t* msg, error_t error);

// Utilisation
call Radio.send(packet); // Revient immédiatement (ou erreur)
// ... CPU peut faire autre chose, ou dormir ...
event Radio.sendDone(packet, SUCCESS) {
printf("Paquet envoyé !");
}

Comparaison : Synchrone vs Asynchrone

CaractéristiqueSynchroneAsynchrone (TinyOS)
BloquageOui (appel attend la fin)Non (revient immédiatement)
Utilisation pileÉlevée (fonction appelle fonction appelle...)Faible (appels courts, pas d'emboîtement)
Efficacité énergétique❌ Mauvaise (CPU actif en attente)✅ Excellente (CPU peut dormir)

TinyOS force l'asynchrone pour économiser l'énergie. C'est pourquoi tout est split-phase : appel → retour immédiat → événement quand c'est fini.

🆕 8. Autres OS IoT modernes — Zephyr et FreeRTOS

Paysage des OS pour IoT

🧒 Au-delà de TinyOS et Contiki

TinyOS et Contiki sont excellents pour la recherche et le prototypage, mais l'industrie exige de la flexibilité. Aujourd'hui, Zephyr (Linux Foundation), FreeRTOS (Amazon) et RIOT OS dominent le marché.

Zephyr OS

🐧 Zephyr OS — Le futur de l'IoT open-source

Créateur : Linux Foundation | License : Apache 2.0 | Statut : Projet actif majeur

Architecture modulaire : Utilise Kconfig (comme Linux) pour configurer les fonctionnalités. Tu peux enlever ce qui ne sert pas → très léger.

Support matériel : 200+ boards (ESP32, Nordic nRF5, STM32, Raspberry Pi Pico, etc.)

Protocoles : Bluetooth, Bluetooth LE, WiFi, LoRa, NB-IoT — tous intégrés nativement.

Multitâche : Threads préemptifs avec priorités. Semaphores, mutexes, queues pour synchronisation. Bien plus haut niveau que TinyOS.

Cas d'usage : Produits commerciaux, prototypage professionnel, standards industriels (Thread, Matter).

💡 À retenir : Zephyr devient le standard de facto pour l'IoT. Ressemble à une version allégée de Linux.

FreeRTOS

⚙️ FreeRTOS — Le RTOS du marché

Créateur : Amazon Web Services | License : MIT | Statut : Leader industrie

Philosophie : RTOS = Real-Time Operating System. Threads avec priorités, switching temps-déterministe, préemption.

Modèle : Tasks (fils d'exécution) avec priorités. Scheduler préemptif (plus haut = plus urgent). Primitives : semaphores, mutexes, message queues.

Portabilité : 40+ architectures CPU (ARM, RISC-V, x86, MIPS, etc.). Code très portable, peu de tuning matériel.

Cas d'usage : Applications critiques en temps-réel (médicaux, automotive). Aussi utilisé pour prototyper Zephyr.

Mémoire : ~6 KB de RAM minimum pour kernel + une task. Très efficace.

💡 À retenir : Si tu veux un vrai RTOS avec garanties temps-réel, FreeRTOS est le choix industriel.

Mbed OS

🔵 Mbed OS — La solution ARM

Créateur : ARM Holdings | License : Apache 2.0 | Cas d'usage : Prototypage rapide

Particularité : API en C++ (plus haut niveau que TinyOS/Contiki). Threads, événements, périphériques abstraits.

Hardware : Optimisé pour processeurs ARM (Cortex-M, Cortex-A). Bon support des cartes de développement.

Trend : Moins utilisé que Zephyr. Beaucoup de projets ARM migrent vers Zephyr pour plus de flexibilité.

Tendances actuelles

📌 Vers où va l'industrie ?

Zephyr en ascension : Soutenu par Linux Foundation + gros contributeurs (Intel, Nordic, NXP). Devient le standard de facto. Nouveaux projets choisissent Zephyr.

FreeRTOS stabil : Dominatrice en aerospace/médical. Amazon y investit. Restera leader pour temps-réel critique.

TinyOS/Contiki décline : Restent excellents pour recherche. Moins attrayants pour produits commerciaux (API vieillies, moins de hardware).

RIOT OS émergent : Gagne en visibilité. Très léger, bon pour applications contraintes (5G edge, smart city).

Pour un nouveau projet IoT en 2025 : Zephyr si tu veux production/standard, FreeRTOS si tu veux du temps-réel pur, TinyOS/Contiki si tu dois intégrer du code 20 ans old.

🌐 9. Piles réseau légères pour systèmes embarqués

🧒 Comment faire du réseau avec 8 KB de RAM ?

Sur un PC, la pile TCP/IP de Linux occupe plusieurs mégaoctets. Un capteur IoT n'a que quelques kilobytes de RAM. Impossible d'utiliser une pile TCP/IP standard — il faut des implémentations ultra-allégées.

uIP — La pile IPv4 minimaliste

📘 Théorie

uIP ("micro IP") est une pile IPv4 conçue pour les systèmes embarqués les plus contraints. Elle occupe seulement 4–5 KB de code et quelques centaines d'octets de RAM.

Comment est-ce possible ? Par des choix de conception très restrictifs :

  • Un seul buffer de paquets : plus facile à gérer, moins de mémoire. L'application doit traiter le paquet immédiatement (ou le copier)
  • TCP simplifié : un seul segment TCP "en vol" à la fois — le prochain paquet attend l'ACK. Pas de fenêtre glissante, pas de contrôle de congestion
  • Retransmissions TCP : c'est à l'application de recréer le paquet perdu (si le paquet contenait des données de capteur, on peut simplement envoyer de nouvelles données)
  • Un seul buffer de réassemblage IP : un seul réassemblage de fragment à la fois
uIP = compatible mais pas performant. Il implémente le minimum pour être conforme au standard, pas plus.

API événementielle dans Contiki (uIP)

📘 Théorie

L'utilisation de uIP dans Contiki est basée sur des événements, comme tout le reste :

while(1) { PROCESS_YIELD(); if(ev == tcpip_event) { // vérifier la raison de l'événement : // paquet entrant, connexion entrante, retransmission, etc. if(uip_newdata()) { ... } } }

📘 Principe de conception

Ces piles légères suivent toutes le même principe : "N'implémenter que ce qui est nécessaire pour la conformité au standard". Elles sont compatibles mais pas nécessairement performantes. Le compromis est acceptable pour les capteurs IoT qui n'envoient que de petites quantités de données à intervalles réguliers.
