Chapitre 9 : 6LoWPAN

6LoWPAN = IPv6 over Low-power Wireless Personal Area Networks. C'est la "colle" qui permet à IPv6 de fonctionner sur 802.15.4. Le défi : IPv6 a besoin de 1280 bytes minimum (MTU), mais 802.15.4 ne fait que 127 bytes ! La solution : compression d'en-tête et fragmentation intelligente.

1. Le problème : IPv6 vs 802.15.4

🚨 Conflit de tailles

IPv6 : En-tête de 40 bytes minimum, MTU de 1280 bytes.

802.15.4 frame : Max 127 bytes total.

Détail :

  • PHR: 1 byte (longueur du frame)
  • 802.15.4 header (MHR): jusqu'à 25 bytes
  • Payload disponible: 127 - 1 - 25 = 101 bytes
  • IPv6 header: 40 bytes
  • Données restantes: 101 - 40 = 61 bytes pour les données !
Chiffres réels

Quand on ajoute UDP (8 bytes) et CoAP (4 bytes minimum) :

Données utiles = 127 - 25 - 40 - 8 - 4 = 50 bytes seulement !

IPv6 exige MTU ≥ 1280 bytes, donc:

Minimum 13 fragments 802.15.4 pour un seul paquet IPv6 ! 😱

2. Solution : 6LoWPAN

✅ Trois techniques

1. Compression d'en-tête (IPHC) : Réduire l'en-tête IPv6 de 40 bytes à quelques bytes.

2. Fragmentation : Fragmenter les paquets IPv6 grands en frames 802.15.4 numérotées.

3. Mesh Addressing : Routage maillé avec en-têtes abrégés.

3. Compression d'en-tête IPHC

Le secret magique
💡 Idée clé : Si le destinataire et la source dérivent leur adresse IPv6 de l'adresse MAC 802.15.4 (qui est déjà dans le frame), pourquoi transmettre l'adresse IPv6 complète ? On peut omettent simplement les 128 bits de l'adresse source et du destinataire !

Compression d'en-tête : cas idéal

Sans compression : 40 bytes (IPv6 header)

Avec compression IPHC (cas optimal) : 2-7 bytes
  • Version : Implicite (IPv6 = 6)
  • Traffic Class : Dérivé ou omis (0 = default)
  • Flow Label : Omis (0 = default)
  • Payload Length : Omis (802.15.4 header indique)
  • Next Header : Implicite (UDP est default en 6LoWPAN)
  • Hop Limit : Souvent 255 ou omis
  • Adresses : Dérivées du MAC 802.15.4 (omises !)
Champ Avec IPv6 Avec IPHC Gain
Version + Traffic Class + Flow Label + Payload Length + Next Header + Hop Limit 14 bytes 1 byte (Dispatch) -13 bytes
Adresse source (128 bits) 16 bytes 0 bytes (dérivée du MAC) -16 bytes
Adresse destination (128 bits) 16 bytes 0 bytes (dérivée du MAC) -16 bytes
TOTAL 40 bytes 2-7 bytes -33 à -38 bytes !

✅ Résultat : Avec IPHC, l'en-tête passe de 40 bytes à seulement 2-7 bytes ! Données disponibles : 127 - 25 - 7 = 95 bytes au lieu de 61 bytes.

4. Fragmentation 6LoWPAN

Découper en morceaux
✂️ Analogie : Comme découper une longue lettre en plusieurs enveloppes numérotées. À la réception, on numérote les enveloppes pour les remettre dans l'ordre et reconstruire la lettre complète.

Format d'en-tête de fragmentation

Fragment header : 4 bytes (minimum)

  • Dispatch byte : 11000xxx (indique "fragment")
  • Datagram size : 11 bits (taille totale en bytes)
  • Datagram tag : 16 bits (identifiant unique pour ce datagram)
  • Datagram offset : 8 bits (position dans le datagram en unités de 8 bytes)

🎬 Démo animée : Fragmentation 802.15.4

Un paquet IPv6 de 300 bytes est fragmenté en 3 frames 802.15.4 (127, 127, 46 bytes). Chaque frame a un numéro de séquence (offset).

5. Dispatch Byte et Stacked Headers

Structure modulaire 6LoWPAN

Les en-têtes 6LoWPAN peuvent être imbriqués. Le Dispatch byte (1 byte) à le début indique quel en-tête suit :

✅ 6LoWPAN est essentiel pour l'IoT. Sans lui, on ne pourrait pas utiliser IPv6 sur 802.15.4. Avec lui, les capteurs peuvent utiliser le même protocole Internet que les ordinateurs (IP end-to-end).

6. Route-over vs Mesh-Under

🧒 Deux façons de faire du multi-hop

Dans un réseau maillé 6LoWPAN, un paquet doit souvent traverser plusieurs nœuds pour atteindre sa destination. Il existe deux approches : faire le routage à la couche IP (Route-over) ou à la couche 6LoWPAN (Mesh-Under).

🌐 Route-over

  • Routage à la couche IP (couche 3)
  • Chaque saut = un saut IP (IPv6)
  • Utilise un protocole de routage IP (ex: RPL)
  • Chaque nœud intermédiaire possède une adresse IPv6
  • Fragment + réassemble à chaque saut IP
  • Plus complexe, mais intégration IPv6 native

🔗 Mesh-Under

  • Routage à la couche 6LoWPAN (couche 2)
  • Un seul saut IP de source à destination
  • Utilise des adresses MAC pour le routage intermédiaire
  • Transparent pour la couche IP
  • Mesh Addressing Header nécessaire (source et destination finales)
  • Moins de surcharge mais routage limité aux adresses 802.15.4

🔧 8. Limites de HC1 et LOWPAN_IPHC (RFC 6282)

⚠️ Limitations de HC1

HC1+HC2 peut compresser les en-têtes IPv6+UDP à seulement 7 octets dans le meilleur cas. Mais ce meilleur cas suppose une communication locale unicast où toutes les informations d'adresse sont déjà dans la trame 802.15.4.

L'avantage est perdu dans ces situations :

  • Communication avec un serveur hors du réseau 6LoWPAN (serveur cloud) → adresse routable complète nécessaire (128 bits)
  • Utilisation d'adresses multicast IPv6
📘 LOWPAN_IPHC — La solution (RFC 6282)

LOWPAN_IPHC remplace HC1 et utilise 2 octets (011 xxxxx yyyyyyyy) pour encoder les options de compression. Les améliorations clés :

  • Compression d'adresses plus flexible : peut compresser des adresses globales en utilisant le contexte partagé entre nœuds
  • Compression d'adresses multicast : supporte plusieurs formes compressées (FF02::00xx → 8 bits, etc.)
  • Compression avec état (stateful) : les nœuds partagent des "contextes" = préfixes d'adresse fréquemment utilisés

Compression stateful : Le protocole Neighbor Discovery en 6LoWPAN (RFC 6775) permet de distribuer des informations de contexte entre nœuds. Un contexte = un préfixe d'adresse de 0 à 128 bits. Il peut référencer des adresses locales ou distantes et a une durée de vie limitée. On peut avoir jusqu'à 16 contextes différents (identifiés par un CID de 4 bits).

Cas optimal : Communication link-local → en-tête IPv6 compressé de 40 octets à 2 octets seulement !

🕸️ 9. En-tête d'adressage maillé (Mesh Addressing Header)

📘 Théorie

Les trames IEEE 802.15.4 ne peuvent être envoyées qu'en un seul saut. Pour le routage Mesh-Under (couche 2), un paquet doit porter deux niveaux d'adresses :

  • L'en-tête 802.15.4 contient l'adresse lien du prochain saut
  • Le Mesh Addressing Header contient les adresses lien de la source originale et de la destination finale

Pourquoi stocker la source/destination originale ? Parce que l'en-tête IP compressé (IPHC) référence les adresses MAC pour la compression — si on ne garde que l'adresse du saut courant, la décompression de l'en-tête IP devient impossible.

En contraste, avec le routage Route-Over (niveau 3), chaque nœud intermédiaire réassemble le paquet IP complet et le refragmente, ce qui est plus coûteux en ressources mais plus simple conceptuellement.

💡 À retenir : Mesh-Under = routage invisible par dessus 802.15.4. Route-Over = routage IP standard avec réassemblage à chaque saut.
