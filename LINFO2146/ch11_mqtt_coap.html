Chapitre 11 : MQTT & CoAP

MQTT et CoAP sont les deux protocoles applicatifs majeurs pour l'IoT. HTTP est trop lourd (700+ bytes d'en-têtes). MQTT et CoAP offrent de légères alternatives : MQTT utilise une architecture publisher-broker-subscriber, CoAP suit le modèle REST d'HTTP mais en binaire compact sur UDP.

1. REST et HTTP

Fondations du Web

REST = Representational State Transfer. Un style architectural basé sur HTTP avec 4 verbes : GET, POST, PUT, DELETE.

🌐 Analogie : HTTP c'est le langage du Web. REST ajoute de la structure : les ressources ont des URL (ex: /users/123), et les verbes HTTP indiquent les actions (GET = lire, POST = créer, PUT = modifier, DELETE = supprimer).
Verbe HTTP Signification Safe ? Idempotent ? Exemple
GET Lire une ressource ✅ Oui ✅ Oui GET /sensors/temp
POST Créer une ressource ❌ Non ❌ Non POST /logs (crée un nouveau log)
PUT Modifier/remplacer une ressource ❌ Non ✅ Oui PUT /config/interval=60
DELETE Supprimer une ressource ❌ Non ✅ Oui DELETE /logs/old

⚠️ Problème HTTP pour IoT : Une requête HTTP simple avec headers = 700+ bytes. Sur 802.15.4 (127 bytes max), impossible ! D'où CoAP.

2. CoAP (Constrained Application Protocol)

HTTP pour IoT

CoAP = "HTTP for constrained devices". Même concepts (GET/POST/PUT/DELETE, URIs), mais en binaire compact sur UDP.

📦 Analogie : Si HTTP est une lettre détaillée avec beaucoup d'information inutile, CoAP est un texto qui dit la même chose avec 10x moins de caractères.

HTTP/REST

Requête GET :

GET /sensors/temp HTTP/1.1 Host: 192.168.1.1 User-Agent: curl/7.64 Accept: */* Connection: close [700+ bytes d'en-tête]

✗ Problèmes :

  • Texte brut (inefficace)
  • TCP (heavy, pas idéal pour IoT)
  • Énormément d'en-têtes

CoAP

Requête GET :

V T TKL C MID 0 0 0 1 [12] GET /sensors/temp [~20 bytes total]

✓ Avantages :

  • Binaire (compact)
  • UDP (léger, rapide)
  • En-tête de 4 bytes !

Format d'en-tête CoAP (4 bytes fixes)

Champ Bits Signification
Ver 2 Version (toujours 01 = version 1)
Type 2 CON (0) = Confirmable, NON (1) = Non-Confirmable, ACK (2), RST (3)
TKL 4 Token Length (0-8 bytes de token)
Code 8 Classe (3b) + Détail (5b) ex: 0.01=GET, 2.05=Content, 4.04=Not Found
Message ID 16 Identifie la requête (pour dédupliquer)
Code CoAP : Structure c.dd (classe . détail)
  • 0.xx = Requête (0.01=GET, 0.02=POST, 0.03=PUT, 0.04=DELETE)
  • 2.xx = Succès (2.01=Created, 2.04=Changed, 2.05=Content)
  • 4.xx = Erreur client (4.04=Not Found, 4.05=Method Not Allowed)
  • 5.xx = Erreur serveur (5.00=Internal Server Error)

🔄 Démo interactive : Requête CoAP

Paquet CoAP généré :

Ver: 01 (v1)
Type: CON (00)
Code: GET (00.01)
Message ID: 1234
URI-Path: sensors/temperature
Taille totale: ~40 bytes

3. CoAP Observe (Push)

Extension Observe : subscription pour IoT

Le client envoie GET /sensors/temp avec l'option Observe: 0 (subscribe).

Le serveur répond avec 2.05 Content + Observe: 1 et la valeur initiale.

Chaque fois que la valeur change, le serveur envoie automatiquement une notification au client avec le nouveau "Observe: 2", "Observe: 3", etc.

Avantage : Le client ne doit pas faire de polling ! C'est du push real-time.

4. MQTT (Message Queuing Telemetry Transport)

Pub/Sub model

MQTT = "MQ Telemetry Transport". Architecture publish-subscribe avec un broker central. Très différent de CoAP.

📰 Analogie : MQTT c'est comme un tableau d'affichage (bulletin board). Les capteurs (publishers) écrivent les données sur le tableau. Les applications (subscribers) lisent le tableau. Le serveur MQTT (broker) gère le tableau. Avantage : 1 capteur alimente 1000 applis !

CoAP (Request-Response)

Client A → CoAP Server (GET /sensors/temp) ← Server B

Chaque client demande directement au serveur. Point-à-point synchrone.

MQTT (Pub/Sub)

Publisher → MQTT Broker ← Subscribers

Découplage total. Pub/Sub asynchrone. 1 source, N applis.

Topics MQTT hiérarchiques

Topics sont organisés en hiérarchie avec / : ucl/barb/room10/temperature

Subscribing :

  • ucl/barb/room10/temperature = cette topic exacte
  • ucl/barb/room10/+ = temperature, humidity, pressure, ... (un seul niveau)
  • ucl/barb/# = tout sous ucl/barb (plusieurs niveaux)
  • ucl/# = tout sous ucl (très large)

📌 Wildcards en détail : + et #

Wildcard Nom Règle Exemple subscription Match ✅ No match ❌
+ Single-level wildcard Remplace exactement un niveau de topic home/+/temperature home/room1/temperature
home/kitchen/temperature		 home/floor1/room1/temperature
(2 niveaux, pas 1)
# Multi-level wildcard Remplace zéro ou plusieurs niveaux. Doit être le dernier caractère. home/# home/temperature
home/room1/temp
home/floor1/room1/temp		 Invalide : home/#/temp
(# pas en dernier)
⚠️ Règles importantes :
  • # seul (sans préfixe) = abonnement à TOUS les topics
  • + peut apparaître plusieurs fois : +/+/temperature = tous les topics à 2 niveaux se terminant par /temperature
  • # ne peut apparaître qu'une fois et uniquement à la fin
  • Topics commençant par $ (ex: $SYS/broker/clients) sont réservés au broker — le wildcard # ne les capture PAS

5. QoS MQTT

QoS Level Nom Sémantique Messages de contrôle Usage
QoS 0 At most once Pas de garantie de livraison Aucun (fire and forget) Capteurs de température (perte acceptable)
QoS 1 At least once Au moins une livraison (possibles doublons) PUBLISH → PUBACK Événements importants, certaines pertes acceptables
QoS 2 Exactly once Livraison garantie, pas de doublons PUBLISH → PUBREC → PUBREL → PUBCOMP (4 messages) Transactions financières, données critiques

6. CoAP vs MQTT

Aspect CoAP MQTT
Architecture Client-Server (REST) Pub-Sub (Broker)
Transport UDP (sans connexion) TCP (avec connexion)
En-tête 4 bytes (binaire) 2 bytes minimum (binaire)
Synchrone/Asynchrone Synchrone (Request-Response) Asynchrone (Publish-Subscribe)
QoS Implicite (type de message) Explicite (QoS 0/1/2)
Push Observe option Natif (Publish)
Cas d'usage IoT direct, 802.15.4, capteurs simples Hub IoT, intégration applis multiples
Taille de paquet ~20-50 bytes ~2-10 bytes (header seul)
✅ Règle de pouce :
  • CoAP pour des appareils bas débit parlant directement à un serveur ou entre eux (réseaux maillés 6LoWPAN).
  • MQTT pour des architectures d'entreprise où les données des capteurs alimentent plusieurs applications (intégration complexe).
  • HTTP(S) pour les applications mobiles et web (navigateur).

7. CoAP — Types de messages et fiabilité

Les 4 types de messages CoAP (champ T)

CoAP définit 4 types de messages encodés dans le champ T (2 bits) de l'en-tête :

Type Valeur T Rôle ACK requis ?
CON (Confirmable) 0 Message fiable. Le destinataire doit répondre par ACK ou RST. ✅ Oui
NON (Non-Confirmable) 1 Message léger sans garantie. Utile pour lectures répétées de capteurs. ❌ Non
ACK (Acknowledgment) 2 Confirme la réception d'un CON. Peut contenir la réponse (piggybacking). —
RST (Reset) 3 Indique que le message CON ou NON a été reçu mais rejeté (token invalide, message inattendu). —

Un CON vide (code 0.00) peut servir de ping : le destinataire répond par RST, confirmant qu'il est en ligne.

Exponential back-off pour les messages CON

Quand un message CON n'est pas acquitté, l'émetteur le retransmet avec un délai exponentiel :

  1. Timeout initial := valeur aléatoire dans [ACK_TIMEOUT, ACK_TIMEOUT × ACK_RANDOM_FACTOR]
  2. Envoyer le message CON, attendre ACK ou RST.
  3. Si pas de réponse et retransmission_counter < MAX_RETRANSMIT : doubler le timeout, incrémenter le compteur, retransmettre.
  4. Abandonner après MAX_RETRANSMIT tentatives.
Valeurs par défaut : ACK_TIMEOUT = 2 s, ACK_RANDOM_FACTOR = 1.5, MAX_RETRANSMIT = 4. Durée max totale ≈ 45 s.

Échange synchrone vs asynchrone

Mode Séquence Avantage / Inconvénient
Synchrone (piggybacking) Client envoie CON → Serveur répond avec ACK contenant directement la réponse ✅ Efficace (2 messages). ❌ Le serveur doit répondre rapidement.
Asynchrone (separate response) Client envoie CON → Serveur répond d'abord avec ACK vide (code 0.00) → puis envoie la réponse dans un nouveau CON ✅ Serveur peut prendre son temps. ❌ 4 messages au lieu de 2. Les tokens permettent d'associer la réponse à la requête initiale.

Les messages CON et NON peuvent être mélangés : ex. requête en NON, réponse en CON, ou l'inverse.

8. CoAP — Sécurité, multicast, découverte et cache

Sécurité CoAP : DTLS et tokens aléatoires

CoAP peut utiliser DTLS (Datagram Transport Layer Security) pour chiffrer et authentifier les échanges, via le schéma coaps:// sur le port 5684.

  • DTLS = version de TLS pour les protocoles datagram (UDP). Protège contre l'espionnage et la falsification.
  • Sans DTLS, les clients doivent utiliser des tokens aléatoires longs pour empêcher les attaques par spoofing : un attaquant envoyant une fausse réponse ne connaît pas le token → la réponse est ignorée par le client.

Multicast et découverte de ressources

CoAP supporte les requêtes vers des groupes multicast IP, ce qui est particulièrement utile dans les réseaux IoT :

  • Les serveurs ne doivent pas envoyer de RST en réponse à un NON-CON multicast non désiré (pour éviter une tempête d'erreurs).
  • Les serveurs doivent attendre un délai aléatoire avant de répondre (éviter la congestion simultanée).

Découverte de ressources (RFC 6690) : Un client peut envoyer GET /.well-known/core à un serveur (ou en multicast). Le serveur répond avec la liste de toutes ses ressources et leurs types. Très utile pour l'intégration automatique d'appareils IoT dans un réseau (M2M).

Caching dans CoAP

CoAP intègre un mécanisme de cache simple, particulièrement utile dans les proxies pour servir les données de nœuds endormis :

  • Option Max-Age : indique la durée de validité du cache (en secondes). Passé ce délai, le proxy revalide la ressource.
  • ETag (Entity Tag) : le serveur inclut un identifiant unique de version dans sa réponse. Le client peut renvoyer cet ETag dans une requête ultérieure → si la ressource n'a pas changé, le serveur répond 2.03 Valid (sans renvoyer le contenu) au lieu de 2.05 Content.
  • If-Match : option utilisée dans les requêtes PUT pour n'écrire la nouvelle valeur que si l'ETag correspond — protection contre les écrasements accidentels concurrents.

9. CoAP — Transfert par blocs et attaques de spoofing

Block-wise transfer (RFC 7959)

UDP ne fragmente pas les messages, et la fragmentation IP est déconseillée sur 802.15.4. Pour accéder à de grandes ressources, CoAP utilise le transfert par blocs :

  • Le client envoie des requêtes GET successives avec une option Block précisant la plage souhaitée : "octets 0–1023", "octets 1024–2047", etc.
  • Chaque bloc est acquitté individuellement si le message est de type CON.
  • Utile pour récupérer des fichiers de configuration, des firmware updates, ou des logs de capteurs.
Exemple : Mettre à jour le firmware d'un nœud IoT en envoyant 20 blocs de 512 octets, chacun acquitté séparément.

Attaques de spoofing CoAP

CoAP tourne sur UDP, sans handshake ni numéros de séquence TCP. Cela ouvre plusieurs vecteurs d'attaque :

Attaque Mécanisme Défense
Spoofing de réponse Un tiers envoie une fausse réponse à la place du vrai serveur Tokens aléatoires longs (le tiers ne connaît pas le token) ou DTLS
Spoofing de RST Un tiers envoie un faux message RST pour interrompre une communication en cours DTLS
Amplification/Reflection Un attaquant usurpe l'IP d'une victime et envoie des requêtes multicast CoAP → la victime reçoit une avalanche de réponses Limiter les réponses multicast, filtrage réseau

10. MQTT — Last Will, messages retenus et MQTT-SN

Last Will & Testament (LWT)

Le Last Will permet au publisher de configurer, lors de la connexion (CONNECT), un message qui sera automatiquement envoyé par le broker à ses subscribers si la connexion est perdue de manière inattendue.

  • Keep-Alive : le client envoie périodiquement un message PINGREQ. Si le broker ne reçoit aucun message du client pendant keep_alive × 1.5 secondes, il considère le client comme déconnecté et diffuse le Last Will.
  • Exemple : sensor/status → "OFFLINE" — les autres appareils savent immédiatement que le capteur est tombé.
  • Exception : si le client se déconnecte proprement avec un message DISCONNECT, le Last Will n'est pas envoyé.

Messages retenus (Retained Messages)

Par défaut, le broker transfère un message uniquement aux subscribers déjà abonnés au moment de la publication. Un subscriber qui s'abonne plus tard ne reçoit rien.

En marquant un message comme Retained (flag dans le PUBLISH), le broker le stocke et l'envoie immédiatement à tout nouveau subscriber.

Exemple : Un capteur de température publie sa dernière valeur comme message retenu. Une nouvelle application qui s'abonne reçoit instantanément la dernière valeur connue, sans devoir attendre la prochaine publication.

MQTT-SN — MQTT pour les réseaux de capteurs

MQTT classique repose sur TCP et utilise les noms de topics en UTF-8, ce qui est trop lourd pour les réseaux 802.15.4. MQTT-SN (MQTT for Sensor Networks) adapte MQTT aux réseaux contraints :

Caractéristique MQTT MQTT-SN
Transport TCP (ports 1883/8883) UDP (sans connexion, plus léger)
Nom de topic Chaîne UTF-8 (variable) ID numérique sur 2 octets ou 1 octet prédéfini
QoS spécial — QoS -1 : publish sans connexion préalable (idéal pour nœuds qui dorment souvent)
Gateway — Passerelle MQTT-SN ↔ MQTT classique obligatoire
Nœuds endormis Non géré nativement La gateway stocke les messages pendant le sommeil du nœud

Architecture : IoT device ←[MQTT-SN/UDP]→ Gateway ←[MQTT/TCP]→ Broker. Les clients peuvent enregistrer leurs noms de topics auprès de la gateway, qui leur attribue un ID court.

✅ CoAP et MQTT couvrent deux paradigmes complémentaires : CoAP (REST over UDP, fiabilité optionnelle, adapté aux réseaux 802.15.4) et MQTT (pub/sub over TCP, Last Will, retained messages, MQTT-SN pour l'IoT contraint). La sécurité passe par DTLS pour CoAP et TLS/SSL pour MQTT.
