Chapitre 12 : LoRa & LoRaWAN

LoRa et LoRaWAN répondent à un besoin critique : les communications longue portée et basse consommation pour l'IoT. Alors que les technologies courte portée (802.15.4, WiFi) ne dépassent pas 100m et les réseaux mobiles (UMTS, LTE) consomment énormément d'énergie et sont coûteux, LoRa propose une alternative révolutionnaire.

1. Pourquoi LoRa ? Le vide technologique

Contexte

En 2010-2015, il existait un vide dans le paysage des technologies IoT : aucune solution n'était vraiment adaptée pour des capteurs longue portée (km), basse consommation (années de batterie), et à bas coût.

🌍 Analogie : 802.15.4 et WiFi, c'est comme avoir un téléphone qui fonctionne bien dans la maison (100m max), mais dès qu'on sort de chez soi, plus rien. UMTS/LTE, c'est comme avoir un téléphone satellite très puissant, mais il décharge la batterie en quelques heures et coûte très cher. LoRa, c'est le bon équilibre : une portée de km avec une batterie qui tient des années.
Technologie Portée Consommation Coût Cas d'usage
802.15.4 (ZigBee) 100 m Très basse Bas Domotique courte portée
WiFi 100 m Moyenne-haute Bas Streaming, données hautes vitesses
GSM 35 km Haute Moyen-élevé Voix téléphonique (en phase de retrait)
UMTS/LTE 5-50 km Très haute Élevé Données mobiles haut débit
LoRa 10+ km Très basse Bas IoT longue portée basse puissance
✅ LoRa comble le vide : longue portée, très basse consommation, bas coût. Parfait pour les réseaux de capteurs distribués qui doivent fonctionner 5-10 ans sur batterie.

2. LoRa ≠ LoRaWAN : Couche physique vs Couche MAC

Distinction cruciale

Attention : LoRa et LoRaWAN ne sont pas la même chose. LoRa est la couche physique propriétaire de Semtech. LoRaWAN est la couche MAC (Medium Access Control) définie par l'Alliance LoRa (open standard).

📡 Analogie : LoRa, c'est le "langage radio" — comment dire les choses sur les ondes. LoRaWAN, c'est la "grammaire et les règles" — qui peut parler, quand, comment on organise la communication. Vous ne pouvez parler que si vous connaissez le langage (LoRa), mais pour que tout le monde se comprenne, il faut aussi les règles (LoRaWAN).
Aspect LoRa (Couche PHY) LoRaWAN (Couche MAC)
Spécification Propriétaire Semtech (documents limités) Open standard (LoRa Alliance, v1.0+)
Modulation Chirp Spread Spectrum (CSS) Utilise LoRa pour transporter les frames
Responsabilité Transmission radio, portée, décodage Accès au médium, authentification, gestion des appareils
Exemple Puce SX1276 (transceiver) Protocole complet avec serveurs réseau et application

3. Architecture typique d'un réseau LoRaWAN

Flux de données

Appareil IoT → Passerelle (Gateway) → Serveur Réseau → Serveur Application → Vos services

  • Appareil IoT (End Device) : Capteur/actionneur avec puce LoRa (ex: Arduino avec SX1276). Envoie ses données vers une ou plusieurs passerelles.
  • Passerelle (Gateway) : Reçoit les signaux LoRa de tous les appareils à proximité, les retransmet au serveur réseau via TCP/IP (WiFi, Ethernet, 4G).
  • Serveur Réseau : Gère les connexions, authentification, déduplication des messages (car plusieurs gateways reçoivent le même message), ordonnancement des réponses.
  • Serveur Application : Reçoit les données décryptées (LoRaWAN déchiffre avec la clé de l'application), exécute la logique métier.

4. Couverture et bandes de fréquence

Portée remarquable

LoRa offre une portée impressionnante : dans les conditions idéales (ligne de vue), plusieurs kilomètres. Même en environnement urbain, une seule gateway peut couvrir toute une ville de taille moyenne.

Exemple réel : LLN (LoRa Lowlands Network) à UCLouvain

Une seule gateway située sur le toit du bâtiment Reaumur couvre aisément la région de Louvain-la-Neuve et ses alentours — plusieurs kilomètres carrés. Cela montre la puissance de la technologie.

5. Modulation LoRa : Chirp Spread Spectrum (CSS)

Technologie clé

La "sauce secrète" de LoRa est son système de modulation : Chirp Spread Spectrum (CSS). C'est ce qui permet la portée longue, la robustesse face aux interférences, et l'orthogonalité entre les spreading factors.

Qu'est-ce qu'un Chirp ?

Un chirp est un signal dont la fréquence varie linéairement avec le temps. Imaginez une sirène qui monte (up chirp) ou descend (down chirp) graduellement.

  • Up chirp : Fréquence augmente linéairement dans le temps. Ex: 863 MHz → 863.125 MHz.
  • Down chirp : Fréquence diminue linéairement.
  • Avantage : Résistant aux interférences bruit blanc. Même si le signal est faible (peu de puissance), les chirps sont reconnaissables.
🎵 Analogie musicale : Si vous fredonnez une note qui monte progressivement (do→ré→mi→...), même si vous chantez doucement, quelqu'un qui vous écoute reconnaîtra la montée. Un son constant qui est faible, vous ne l'entendrez peut-être pas. Le chirp, c'est pareil — la "progression" le rend robuste.

Spreading Factor (SF) : Le grand levier

LoRa utilise des Spreading Factors de SF5 à SF12. Chaque SF divise un symbole en plusieurs chips (pulses de fréquence rapides).

  • SF5 : 32 chips par symbole. Rapide (haut débit), portée courte.
  • SF12 : 4096 chips par symbole. Très lent (bas débit), portée maximale.
  • Règle : SF+1 → symbole 2× plus long, débit 2× plus lent, portée 2-3× meilleure.

Propriété magique : Les SFs sont presque orthogonaux — un récepteur accordé à SF7 peut ignorer les signaux SF8, SF9, etc. qui arrivent sur le même canal/fréquence au même moment. Cela permet multiplexer plusieurs appareils !

Exemple concret 1 : SF7, 125 kHz, CR 4:5

  • Bits par symbole : 7
  • Taux de symboles : 125,000 / 2^7 = 125,000 / 128 = 976.6 symboles/s
  • Coding Rate : 4:5 = 0.8
  • Débit : 7 × 0.8 × 976.6 = 5,467 bit/s ≈ 5.5 kbit/s

Exemple concret 2 : SF12, 125 kHz, CR 4:5

  • Bits par symbole : 12
  • Taux de symboles : 125,000 / 2^12 = 125,000 / 4,096 = 30.5 symboles/s
  • Coding Rate : 4:5 = 0.8
  • Débit : 12 × 0.8 × 30.5 = 292.8 bit/s ≈ 0.29 kbit/s
✅ SF7 @ 125 kHz ≈ 5.5 kbit/s | SF12 @ 125 kHz ≈ 0.3 kbit/s. Le SF divise le débit par deux à chaque incrément, mais augmente la portée et la robustesse.

7. LoRaWAN : Couche MAC et structure des frames

Spécification ouverte

LoRaWAN est défini par la LoRa Alliance (Semtech, Orange, Proximus, etc.). C'est une spécification ouverte (version 1.0.x, 1.1.x) qui structure comment les appareils et gateways communiquent.

Paramètre Description
DevEUI 64-bit identifiant unique de l'appareil (IEEE MAC format)
AppEUI / JoinEUI 64-bit identifiant de l'application / du serveur de jointure
DevAddr 32-bit adresse assignée après activation (unique dans le réseau)
Data Rates (DR) DR0 (SF12, 125 kHz) à DR7 (FSK 50 kbit/s). Seuls DR0-DR6 utilisent LoRa ; DR7 = FSK modulé
Frame Counter Compteur 32-bit (ou 16-bit en v1.0) pour éviter les rejeux

Structure d'une frame LoRaWAN

Une frame LoRaWAN montante (device → réseau) :

MHDR | DevAddr | FCtrl | FCnt | Opts | Port | Payload | MIC 1B | 4B | 1B | 2/4B | ? | 1B | ≤ N B | 4B
  • MHDR (1B) : Type de message (Join Request, Unconfirmed Data, Confirmed Data, Downlink, etc.)
  • DevAddr (4B) : Adresse de l'appareil (assignée lors de l'activation)
  • FCtrl (1B) : Options (ACK, Class bits, FPending, etc.)
  • FCnt (2-4B) : Compteur de frames (protection anti-rejeux)
  • Opts (variable) : Commandes MAC (ne pas confondre avec MQTT)
  • Port (1B) : Numéro de port applicatif (0=MAC, 1-223=application, 224-255=réservé)
  • Payload (0-242B) : Données applicatives (déchiffrées avec AppSKey)
  • MIC (4B) : Message Integrity Code (AES-128 avec NwSKey)
Data Rate (DR) Modulation Débit nominal Taille payload max (UE)
DR0 LoRa SF12, 125 kHz 250 bit/s 51 B
DR1 LoRa SF11, 125 kHz 440 bit/s 51 B
DR2 LoRa SF10, 125 kHz 980 bit/s 51 B
DR3 LoRa SF9, 125 kHz 1760 bit/s 115 B
DR4 LoRa SF8, 125 kHz 3125 bit/s 242 B
DR5 LoRa SF7, 125 kHz 5470 bit/s 242 B
DR6 LoRa SF7, 250 kHz 11000 bit/s 242 B
DR7 FSK 50 kbit/s 50000 bit/s 242 B

8. Accès au médium : ALOHA et retransmissions

Protocole simple

LoRaWAN n'utilise pas de CSMA/CA complexe (comme WiFi). À la place, il utilise ALOHA : les appareils envoient simplement leurs données, et le serveur gère les collisions et retransmissions.

ALOHA : Accès aléatoire simple

Un appareil peut envoyer à tout moment (pas d'écoute du canal avant transmission). Si deux appareils envoient exactement au même moment sur le même canal → collision, les deux messages sont perdus. Le serveur détecte l'absence de réponse et demande une retransmission.

  • Mode Confirmed : L'appareil attend un ACK du serveur. S'il ne le reçoit pas dans la fenêtre RX2, il retransmet après délai aléatoire.
  • Mode Unconfirmed : L'appareil envoie une fois et oublie. Pas de retransmission.
  • Avantage : Très simple, peu d'énergie (pas d'écoute). La fiabilité est déléguée au serveur.
  • Inconvénient : Pas garantie de livraison (c'est au serveur application de gérer). Si le réseau est encombré, perte de messages.

9. Classes d'appareils LoRaWAN (A, B, C)

Trois modèles

LoRaWAN définit trois classes d'appareils, chacune avec un modèle de réception différent. Le choix dépend du cas d'usage : peut-on garder le réseau toujours actif ou doit-il dormir pour économiser l'énergie ?

Classe A : Tous les appareils doivent la supporter

Comportement : L'appareil envoie à tout moment. Après chaque transmission, il ouvre 2 fenêtres de réception :

  • RX1 : 1-5 secondes après transmission. Même canal et débit que l'uplink.
  • RX2 : 2 secondes après RX1 fermeture (ou après RX1). Canal et DR préconfigurés, généralement SF12 @ 869.525 MHz (UE).

Propriété clé : Le serveur peut UNIQUEMENT envoyer une réponse (downlink) dans ces fenêtres. En dehors, impossible.

Cas d'usage : Capteurs qui envoient rarement (quelques fois par jour) → dorment 99% du temps → batterie dure des années.

Exemple durée de vie : Capteur envoie 1× par jour, 50 bytes, SF7 @ 125 kHz ≈ 46 ms par transmission. Avec deux RX windows → énergie négligeable. Batterie AA → 5-10 ans.

Classe B : Écoute programmée (Beacon)

Amélioration par rapport à A : L'appareil écoute périodiquement un beacon (signal de synchronisation envoyé par le serveur), qui définit des "ping slots" où il peut recevoir des messages du serveur.

  • Période beacon : Par défaut 128 secondes.
  • Ping slots : 4096 par période. Durée par slot : (128 - 2.12 - 3) / 4096 ≈ 30 ms.
  • Avantage : Le serveur peut atteindre l'appareil à des moments prévisibles (sans attendre qu'il envoie).
  • Inconvénient : L'appareil doit écouter toutes les 128/4096 ≈ 31 ms → consommation plus élevée.

Cas d'usage : Appareils nécessitant une latence modérée (lampadaires qu'on peut éteindre à distance, etc.). Batterie : quelques mois à 1 an.

Classe C : Réception continue (mais alternée)

Comportement : Après chaque transmission, l'appareil ouvre RX1 et RX2. Entre deux transmissions, il garde le RX2 toujours ouvert (ou les deux canaux alternativement).

  • Propriété : Quasiment latence zéro. Le serveur peut envoyer un downlink presque instantanément.
  • Inconvénient : Récepteur allumé presque tout le temps → très consommateur d'énergie.
  • Cas d'usage : Appareils sur secteur (pas de batterie) : lampadaires intelligents, routeurs, serveurs LoRa.
Classe Latence downlink Consommation Durée batterie Cas d'usage
A 1-5 + 1 sec (après uplink) Très basse 5-10 ans Capteurs qui envoient rarement
B ≤ 128 sec (programmé) Basse-modérée 6-12 mois Appareils avec latence tolérée
C Quasi-immédiate Haute (secteur) — Appareils sur secteur

10. Temps en l'air (Time on Air)

Limitation importante

Chaque transmission occupe le canal pendant un certain temps. En Europe, la réglementation limite l'utilisation à 1% du temps (ou moins selon la bande). Cela limite le nombre de messages/jour.

Calcul du Time on Air

Le Time on Air dépend du SF, de la largeur de bande, du CR, et de la taille du payload.

Formule approximative :

ToA ≈ (Nombre de symboles) × (Durée d'un symbole) Durée d'un symbole = 2^SF / Bandwidth

Exemples concrets

  • SF7, 125 kHz, CR 4:5, 1 byte payload : ToA ≈ 46.3 ms
  • SF12, 125 kHz, CR 4:5, 1 byte payload : ToA ≈ 1155.1 ms (plus d'1 seconde !)
  • SF12, 125 kHz, CR 4:5, 20 byte payload : ToA ≈ 2400+ ms

Implication : Avec un duty cycle 1% en UE, un appareil peut envoyer :

  • SF7 @ 20 bytes : ~51 messages/jour (86,400 s × 1% / 46 ms)
  • SF12 @ 20 bytes : ~2 messages/jour

11. Réglementation Duty Cycle en Europe

L'UE divise la bande 863-870 MHz en sous-bandes avec des restrictions duty cycle différentes :

Sous-bande Fréquence Duty Cycle max
K 863.0-865.0 MHz 0.1%
L 865.0-868.0 MHz 1%
M 868.0-868.6 MHz 1%
N 868.7-869.2 MHz 0.1%
P 869.4-869.65 MHz 10% (Bande "libre")
Q 869.7-870.0 MHz 1%
✅ Respecter le duty cycle est obligatoire. Les devices et gateways doivent compter leur temps d'émission et interrompre si le seuil est atteint.

12. Passerelles (Gateways) LoRaWAN

Point central

Une gateway LoRa est un équipement intermédiaire qui reçoit les signaux LoRa de tous les appareils et les retransmet au serveur réseau via IP (WiFi, Ethernet, 4G).

Aspect Appareil LoRa (End Device) Gateway LoRa
Transceiver SX1276, SX1272 (1 canal) SX1302, SX1308 (multicanal, jusqu'à 10 ch simultanés)
Puissance TX +14 dBm typical +27 dBm typical
Réception Un message à la fois Plusieurs messages simultanés sur différents canaux/SFs
Coût €5-20 (module) €100-500+ (équipement complet)
Déploiement Intégré dans capteurs/appareils Installé en hauteur (toit, mât) pour couvrir une région

Capacité d'une gateway

Une gateway moderne (SX1302) peut traiter environ :

  • Réception simultanée : ~10 canaux × SF12 (worst case).
  • Débit de messages : Typiquement ~50,000 messages/jour avec payload moyen 20 bytes (sous duty cycle 1%).
  • Dans des conditions optimales : jusqu'à 700,000 messages/jour.

13. Déduplication et prévention des rejeux

Déduplication par le serveur réseau

Comme plusieurs gateways reçoivent le même message (si l'appareil est à la limite de la portée de plusieurs gateways), le serveur réseau déduplique en utilisant :

  • Frame Counter : Chaque message a un compteur croissant. Si un message arrive deux fois avec le même compteur → doublon → ignorer.
  • RSSI + timestamp : Si deux messages avec le même compteur arrivent à des moments différents ou avec des forces de signal différentes → le meilleur est gardé.
  • Prévention des rejeux : Si le compteur n'a pas augmenté depuis la dernière réception → message de rejeu → rejeter.

14. Sécurité LoRaWAN : Clés et chiffrement

Authentification et confidentialité

LoRaWAN utilise AES-128 pour l'authentification et le chiffrement. Deux clés sont impliquées :

Clé Taille Utilisée par Fonction
NwSKey (Network Session Key) 128 bits Appareil + Serveur réseau Calcule le MIC (Message Integrity Code). Protège contre les modifications.
AppSKey (Application Session Key) 128 bits Appareil + Serveur application Chiffre/déchiffre le payload applicatif (AES-128-CTR).

Séparation des responsabilités

C'est une architecture intéressante :

  • Serveur réseau : Reçoit le message avec MIC. Peut vérifier que le message n'a pas été modifié (avec NwSKey), mais ne peut pas lire le payload (chiffré avec AppSKey).
  • Serveur application : Reçoit le payload déchiffré. Il ne connaît pas NwSKey (donc ne peut pas vérifier le MIC lui-même), mais c'est le serveur réseau qui l'a garanti.

15. Activation des appareils : OTAA et ABP

Deux approches

Avant de pouvoir communiquer, un appareil doit être "activé" : il faut que le réseau lui assigne une adresse (DevAddr) et des clés de session.

OTAA (Over-The-Air Activation)

Principe : L'appareil demande son activation via un "Join Request".

  • Avant : L'appareil a DevEUI, AppEUI, AppKey (stockés en usine ou en manufacture).
  • Join Request : Message signé avec AppKey contenant DevEUI + AppEUI + DevNonce (aléatoire).
  • Join Accept : Le serveur répond avec JoinNonce (aléatoire), assignant : DevAddr, NwSKey, AppSKey.
  • Après : L'appareil connaît ses clés de session et peut communiquer.

Avantage : AppKey jamais transmis en clair. Sécurisé. Idéal pour les appareils vendus en masse (chacun a sa clé).

Note v1.0.4+ : Depuis v1.0.4, il y a aussi NwkKey (clé réseau séparé) et un "Join Server" dédié.

ABP (Activation By Personalization)

Principe : Les clés sont préconfigurées manuellement sur l'appareil avant déploiement.

  • L'administrateur réseau configure : DevAddr, NwSKey, AppSKey directement sur l'appareil (via UART, configuration file, etc.).
  • L'appareil envoie des messages immédiatement, sans Join Request.
  • Avantage : Plus rapide, pas besoin de serveur Join. Simple pour prototypage.
  • Inconvénient : Si on change de réseau (ex: Proximus → The Things Network), faut reconfigurer tous les appareils. Clés moins sécurisées (partagées, potentiellement vue par plusieurs personnes).
Aspect OTAA ABP
Configuration préalable DevEUI, AppEUI, AppKey DevAddr, NwSKey, AppSKey
Join process Dynamique (chaque démarrage) Aucun (clés directement utilisées)
Sécurité Haute (AppKey jamais transmis) Plus faible (clés préfixées, moins flexibles)
Portabilité Haute (changement de réseau facile) Faible (reconfiguration nécessaire)
Cas d'usage Production, multi-réseau Prototypage, réseau unique

16. Construire un réseau LoRaWAN

Options pratiques

Vous avez plusieurs options pour déployer LoRaWAN : acheter du matériel, rejoindre un réseau existant, ou utiliser un service commercial.

Option 1 : Infrastructure privée

Acheter une gateway (ex: Kerlink, Semtech, Huawei) + serveur réseau (ChirpStack, The Things Stack privé).

  • Coût : Gateway €200-1000+, serveur peut être un Raspberry Pi.
  • Avantage : Entièrement contrôlé, aucune dépendance externe.
  • Inconvénient : Vous gérez tout (mises à jour, sécurité, disponibilité).

Option 2 : The Things Network (TTN) - Gratuit

TTN est un réseau LoRaWAN public et gratuit, soutenu par The Things Industries et des bénévoles.

  • Coût : Gratuit ! (Fair use policy : 30 sec/jour/device, payload limité).
  • Avantage : Infrastructure mutualisée. Pas de maintenance. Idéal pour prototypage et petits projets.
  • Inconvénient : Vous dépendez de bénévoles. Limite 30 sec/jour/device. Pas de garantie SLA.

Option 3 : Services commerciaux

Orange, Proximus (Belgique), KPN (Pays-Bas), etc. offrent des services LoRaWAN gérés.

  • Proximus (Belgique) :
    • Plan "Heavy" : ~€1/appareil/mois.
    • Plan "Light" : €0.50 + coût par message.
  • Avantage : SLA garanti, infrastructure nationale, support professionnel.
  • Inconvénient : Coûteux pour beaucoup d'appareils (mais rentable à grande échelle).

Résumé pour l'examen

Concepts clés à retenir

  • LoRa PHY : Modulation Chirp Spread Spectrum. SF5-SF12. Plus SF = plus loin mais plus lent.
  • LoRaWAN MAC : Spécification ouverte. DevEUI, AppEUI, DevAddr. Frames avec header + payload + MIC.
  • Débit : SF7 @ 125 kHz ≈ 5.5 kbit/s. SF12 ≈ 0.3 kbit/s. Calculez avec la formule : Bit Rate = SF × CR × (BW / 2^SF).
  • Classes : A = tous les devices, dormir 99% du temps. B = beacon + ping slots. C = réception continue (secteur).
  • Duty cycle UE : 1% max (ou 0.1% selon la bande). Limite le nombre de messages/jour.
  • Sécurité : NwSKey (MIC) + AppSKey (chiffrement payload). OTAA = dynamique, sûr. ABP = statique, plus simple.
  • Gateway : Reçoit tous les appareils, déduplique via frame counter, retransmet au serveur réseau via IP.
  • Cas d'usage real : Smart meters, capteurs agricoles, suivi d'actifs, monitoring environnemental.

Formules importantes

  • Symbol Rate : BW / 2^SF (symboles/s)
  • Bit Rate : SF × CR × Symbol Rate
  • Durée symbole : 1 / Symbol Rate (ms)
  • Time on Air : ~(Preamble + Header + Payload) × Durée symbole
  • Fair use TTN : 30 sec/jour/device = 30,000 ms

Pièges courants à l'examen

  • Confondre LoRa (PHY) avec LoRaWAN (MAC). Les deux sont nécessaires.
  • Oublier que SF+1 = 2× plus long, donc 2× plus lent en débit, mais 2-3× meilleur en portée.
  • Penser que les appareils peuvent recevoir n'importe quand. Non ! Classe A = seulement RX1/RX2 après uplink.
  • Ignorer le duty cycle. Les appareils doivent rester dans les limites légales.
  • Confondre NwSKey (serveur réseau) et AppSKey (serveur application). Deux clés différentes, deux serveurs différents.
  • Croire que OTAA est gratuit. Joindre consomme du Time on Air aussi !
