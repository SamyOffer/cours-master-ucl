Section 1 — C'est quoi un Pattern Fréquent ?

🧒 C'est quoi un pattern ?

Imagine que tu travailles à la caisse d'un supermarché. Après des milliers de tickets de caisse, tu remarques que les gens qui achètent des pâtes achètent aussi souvent de la sauce tomate. C'est un pattern fréquent !

🛒 Transactions
📋 Itemsets
🔢 Support
🎯 Pattern Fréquent

Définition

Un pattern fréquent est un ensemble d'items (itemset) qui apparaît dans un nombre suffisant de transactions d'une base de données. La notion de "suffisant" est définie par un seuil de support minimum.

💡 À retenir : Les patterns fréquents sont la base de la découverte de règles d'association. Ils représentent les comportements récurrents dans les données.

Section 2 — Base de Données Transactionnelle

Voici un exemple typique de base de données transactionnelle :

Transaction Items
T1 pain, beurre, lait
T2 pain, café, beurre, oeufs
T3 lait, café
T4 pain, lait, café, beurre
T5 café, oeufs

Concepts clés

Item : Un produit individuel (pain, beurre, café, etc.)

Transaction : Un ensemble d'items achetés ensemble

Itemset : Un sous-ensemble arbitraire d'items

k-itemset : Un itemset contenant exactement k items

Mesures Mathématiques

1. Support Count

Signification : Le nombre de transactions qui contiennent l'itemset I

I = itemset

D = base de données

cover = ensemble des transactions couvrant I

Exemple : support_count({pain, beurre}) = 3 (dans T1, T2, T4)
2. Support Relatif

Signification : La proportion de transactions contenant l'itemset

Exemple : support({pain, beurre}) = 3/5 = 0.6 = 60%
3. Fonction Match

Signification : Retourne 1 si la transaction τ contient tous les items de I

4. Cover

Signification : L'ensemble de toutes les transactions contenant I

Section 3 — Règles d'Association

Une règle d'association est une implication de la forme A ⇒ B, où A et B sont des itemsets disjoints.

Confiance d'une Règle

Signification : La probabilité que B soit acheté étant donné que A a été acheté

Exemple : conf({pain} ⇒ {beurre}) = 3/4 = 75%

Section 4 — Algorithme Apriori

💡 L'Idée Clé

Imagine que tu veux trouver tous les groupes d'amis qui se retrouvent souvent dans les mêmes fêtes. La propriété magique : si un groupe de 3 amis est rare, alors tout groupe de 4 amis qui les inclut est aussi rare ! Cela permet d'éliminer des candidats sans les tester.

Propriété d'Anti-Monotonie

Cette propriété fondamentale permet à Apriori de réduire drastiquement le nombre de candidats testés.

Étapes de l'Algorithme Apriori

  1. Trouver tous les 1-itemsets fréquents (L₁)
  2. Générer les candidats 2-itemsets à partir de L₁
  3. Élaguer les candidats en utilisant l'anti-monotonie
  4. Trouver les 2-itemsets fréquents (L₂)
  5. Répéter jusqu'à ce qu'aucun nouveau itemset fréquent ne soit trouvé

Section 5 — FP-Growth Algorithm

🌳 L'Idée Clé

Plutôt que de générer des candidats comme Apriori, FP-Growth compresse la base de données dans une structure arborescente appelée FP-Tree. Les patterns fréquents sont ensuite extraits directement de cet arbre, sans générer de candidats.

Structure de l'Arbre FP

L'algorithme FP-Growth fonctionne en deux passes :

  • Première passe : Compter la fréquence de chaque item
  • Deuxième passe : Construire l'FP-Tree en triant les items par fréquence décroissante

Avantages vs Apriori

Aspect Apriori FP-Growth
Génération de candidats Oui (coûteux) Non
Nombre de scans de la base Élevé (k passes) 2 passes seulement
Mémoire Peut être importante Compression avec FP-Tree
Performance sur grandes bases Lente Rapide
💡 À retenir : FP-Growth est généralement plus efficace qu'Apriori sur les grandes bases de données car il évite la génération coûteuse de candidats.

Section 6 — ECLAT Algorithm

ECLAT (Equivalence Class Cluster Transformation) utilise une représentation verticale de la base de données, contrairement au format horizontal (transaction → items) utilisé par Apriori et FP-Growth.

Format Vertical : TID-lists

Au lieu de stocker (transaction → items), ECLAT stocke (item → liste de transactions) :

Item TID-list Support
pain {T1, T2, T4} 3
beurre {T1, T2, T4} 3
lait {T1, T3, T4} 3
café {T2, T3, T4, T5} 4
oeufs {T2, T5} 2

Calcul du Support par Intersection

Pour calculer le support d'un itemset composé, on fait l'intersection des TID-lists :

Exemple :
TID({pain}) = {T1, T2, T4}
TID({beurre}) = {T1, T2, T4}
TID({pain, beurre}) = TID({pain}) ∩ TID({beurre}) = {T1, T2, T4}
support({pain, beurre}) = |{T1, T2, T4}| = 3
💡 Avantage d'ECLAT : La représentation verticale permet des calculs de support très efficaces par simple intersection d'ensembles.
